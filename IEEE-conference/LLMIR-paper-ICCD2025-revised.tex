\begin{abstract}
Large Language Model (LLM) inference faces significant challenges in memory management, computational efficiency, and deployment scalability. This paper presents LLMIR, a compiler infrastructure based on MLIR (Multi-Level Intermediate Representation) for optimizing LLM inference workloads. Unlike runtime-centric approaches that rely solely on dynamic optimizations, LLMIR enables systematic compile-time analysis of LLM-specific patterns including PagedAttention, KV cache management, and quantization. Our contributions include: (1) IR-level PagedAttention representation for static block-allocation analysis, (2) a four-stage compilation pipeline from PyTorch/ONNX to optimized kernels, and (3) a reproducible evaluation stack (MVP-A/B/C) linking compiler passes to serving benchmarks. \textbf{Verified in open-source CI:} MVP-A rewrites oversized KV block attributes and matches a numeric reference; MVP-B demonstrates ShareGPT-shaped prefix reuse ($\sim$4--5$\times$ KV-layer speedup); MVP-C removes CPU NumPy round-trips via block-paged GPU KV and chained \texttt{past\_key\_values}. \textbf{Projected at scale:} Table~\ref{tab:multi_model} lists design-target throughput on A100-80GB (not yet measured end-to-end by our GPU harness). PPL/MMLU rows in Table~\ref{tab:quality} are illustrative until a quality harness is checked in.
\end{abstract}

\section{Introduction}

The rapid advancement of Large Language Models (LLMs) has revolutionized artificial intelligence applications, from natural language processing to code generation and reasoning tasks \cite{b1}. However, as model sizes continue to grow exponentially, efficient inference has become a critical bottleneck, particularly in production environments where latency, throughput, and resource utilization directly impact user experience and operational costs \cite{b2}.

Current LLM inference frameworks address specific aspects of this challenge through runtime optimizations. vLLM \cite{b3} introduces PagedAttention for memory-efficient KV cache management at runtime, while SGLang \cite{b4} focuses on structured generation and task scheduling. TensorRT-LLM \cite{b10} provides GPU-optimized kernels through ahead-of-time compilation but lacks LLM-specific IR abstractions. MLC-LLM \cite{b15} leverages TVM for cross-platform deployment but operates at a lower abstraction level without specialized PagedAttention support.

These approaches share a fundamental limitation: they lack a unified intermediate representation that captures LLM-specific computation patterns at a level suitable for systematic compiler optimization. Runtime systems make optimization decisions dynamically, missing opportunities for cross-operation analysis, memory layout optimization, and hardware-specific code generation that compilers can provide.

This paper presents LLMIR (Large Language Model Intermediate Representation), a compiler infrastructure built on MLIR \cite{b5} that addresses these limitations. Our key contributions are:

\begin{itemize}
    \item \textbf{LLM-Specific Dialect Design}: A comprehensive MLIR dialect with custom types (PagedKVCache, ShardedTensor, QuantizedTensor) and operations that capture LLM inference semantics, enabling static analysis of previously runtime-only patterns.
    
    \item \textbf{Compiler-Level PagedAttention}: The first IR-level representation of PagedAttention, enabling compile-time analysis of block allocation patterns, access locality, and cross-sequence sharing opportunities.
    
    \item \textbf{Multi-Stage Compilation Pipeline}: A detailed transformation flow from PyTorch/ONNX models through LLM dialect to optimized CUDA kernels, with clear separation of optimization concerns at each level.
    
    \item \textbf{Reproducible MVP Evaluation}: An open-source harness (MVP-A/B/C) linking compiler passes to serving benchmarks---block-size IR rewrites, ShareGPT-shaped prefix reuse, and GPU KV paths---with JSON artifacts checked into the repository (\texttt{IEEE-conference/benchmarks/}).
\end{itemize}

\section{Related Work}

\subsection{LLM Inference Frameworks}

Runtime-based LLM inference systems have made significant advances in memory and compute optimization. vLLM \cite{b3} introduced PagedAttention, managing KV cache through a virtual memory-like paging system that achieves near-zero memory waste. SGLang \cite{b4} optimizes structured generation through RadixAttention and efficient scheduling. These systems make optimization decisions at runtime based on dynamic workload characteristics.

TensorRT-LLM \cite{b10} represents NVIDIA's production solution, providing optimized CUDA kernels compiled ahead-of-time. While it achieves excellent GPU utilization, it operates at the CUDA kernel level without LLM-specific IR abstractions, limiting cross-operation optimization opportunities.

\subsection{Compiler Approaches for Deep Learning}

Deep learning compilers have evolved from general-purpose frameworks to domain-specific solutions. XLA \cite{b13} optimizes TensorFlow graphs through operator fusion and layout optimization. TVM \cite{b14} provides an end-to-end compilation stack with automatic tuning. MLIR \cite{b5} offers an extensible multi-level IR framework adopted by many compilation projects.

MLC-LLM \cite{b15} applies TVM compilation to LLM deployment, enabling cross-platform execution but lacking specialized PagedAttention support and operating primarily at tensor computation level. Torch-MLIR \cite{b16} provides PyTorch-to-MLIR conversion but without LLM-specific optimizations.

LLMIR distinguishes itself by providing: (1) LLM-specific IR abstractions capturing PagedAttention and KV cache semantics, (2) compile-time analysis of patterns previously optimized only at runtime, and (3) a complete compilation pipeline from model import to hardware-specific code generation.

\section{LLMIR Architecture and Design}

\subsection{System Overview and Compilation Flow}

Figure \ref{fig:architecture} illustrates LLMIR's architecture and the complete compilation flow from model input to optimized execution. The system transforms models through four distinct stages:

\textbf{Stage 1 - Model Import}: Frontend converters translate PyTorch, ONNX, or vLLM model graphs into LLMIR's high-level representation. During import, attention patterns are recognized and converted to LLM dialect operations (e.g., \texttt{llm.attention}, \texttt{llm.paged\_attention}).

\textbf{Stage 2 - High-Level Optimization}: The LLM dialect enables optimizations that reason about LLM-specific patterns: KV cache block allocation analysis, attention operation fusion, and cross-sequence sharing detection.

\textbf{Stage 3 - Lowering and Code Generation}: Operations progressively lower through MLIR's standard dialects (linalg, memref, scf) to hardware-specific representations. For CUDA targets, this produces PTX code with optimized memory access patterns.

\textbf{Stage 4 - Runtime Integration}: Generated code integrates with a thin runtime layer handling dynamic batch management and memory allocation. Crucially, the runtime executes pre-compiled, optimized code rather than making optimization decisions dynamically.

\subsection{LLM Dialect Design}

The LLM dialect provides three custom types that capture essential LLM inference data structures with formal semantic invariants:

\textbf{PagedKVCacheType} encapsulates paged KV cache storage:
\begin{mlircode}
\begin{lstlisting}[style=mlir]
!llm.paged_kv_cache<f16, 32, 32, 128, 16, 8192>
// element_type, num_layers, num_heads, 
// head_dim, block_size, max_seq_len
\end{lstlisting}
\end{mlircode}

This type enables the compiler to reason about memory requirements and access patterns statically. The type enforces the following invariants: (1) Memory layout where each block stores $B$ tokens as $\text{Block}[\text{layer}][\text{head}] = \text{tensor}\langle B \times d_h \times \text{dtype}\rangle$; (2) Maximum blocks per sequence $= \lceil L_{max}/B \rceil$; (3) Total KV cache memory bound $\leq 2 \times n_L \times n_H \times n_B \times B \times d_h \times \text{sizeof}(\text{dtype})$.

\textbf{ShardedTensorType} represents tensors partitioned across devices:
\begin{mlircode}
\begin{lstlisting}[style=mlir]
!llm.sharded_tensor<f16, [4096, 4096], 1, 4, 2>
// element_type, shape, shard_dim, 
// num_shards, shard_index
\end{lstlisting}
\end{mlircode}

The type enforces dimension divisibility: $\text{shape}[\text{shard\_dim}] \mod n_s = 0$, and computes shard shape automatically for efficient distributed execution.

\textbf{QuantizedTensorType} captures quantization parameters with safety analysis:
\begin{mlircode}
\begin{lstlisting}[style=mlir]
!llm.quantized_tensor<i4, [4096, 4096], 
  true, true, 1, 128, 4>
// element_type, shape, symmetric, per_channel,
// quant_axis, group_size, num_bits
\end{lstlisting}
\end{mlircode}

The compiler performs quantization safety analysis, computing expected accuracy impact as $\text{error} = \sigma_w / (2^{n-1} - 1)$ where $\sigma_w$ is the weight standard deviation and $n$ is the bit-width.

\subsection{PagedAttention IR Representation}

A key innovation is the IR-level representation of PagedAttention, enabling compile-time analysis of dynamic memory patterns. The representation captures both structural and behavioral aspects:

\begin{mlircode}
\begin{lstlisting}[style=mlir]
// Allocate paged KV cache with static analysis
%kv = llm.create_paged_cache {
  block_size = 16, 
  initial_blocks = 1024,
  growth_factor = 1.5
} : !llm.paged_kv_cache<f16,32,32,128,16,8192>

// Append with compile-time block analysis
%kv2, %blocks = llm.append_kv %kv, %k, %v, %seq 
  {enable_sharing = true, 
   min_blocks = 4, max_blocks = 512} : ...

// PagedAttention with optimized kernel selection
%out = llm.paged_attention %q, %kv2, %blocks 
  {kernel_variant = "flash",
   scale = 0.088388, causal = true} : ...
\end{lstlisting}
\end{mlircode}

The compiler performs three key analyses on these operations:
\begin{enumerate}
    \item \textbf{Block Requirements Prediction}: Given static bounds on sequence length $[L_{min}, L_{max}]$, compute block allocation range $[\lceil L_{min}/B \rceil, \lceil L_{max}/B \rceil]$ and optimize memory pool sizing.
    \item \textbf{Sharing Detection}: Through dataflow analysis and value numbering, identify sequences with common prefixes. When operations share defining operations for key tensors, the compiler marks them for copy-on-write optimization.
    \item \textbf{Kernel Selection}: Based on sequence length bounds and hardware capabilities (shared memory size, compute capability), select optimal attention kernel variants at compile time.
\end{enumerate}

\section{Implementation Details}

\subsection{KV Cache Optimization Pass}

The KVCacheOptimization pass implements pattern-based rewrites that analyze and transform KV cache operations. Algorithm \ref{alg:block_opt} describes the block size optimization logic with formal metrics:

\begin{figure}[htbp]
\small
\begin{algorithmic}[1]
\STATE \textbf{Input:} AppendKVOp $op$, hardware config $H$
\STATE \textbf{Output:} Optimized operation with adjusted block size
\STATE $S \gets$ \{observed sequence lengths from IR analysis\}
\STATE $d_h \gets$ head dimension from type
\FOR{$B \in \{16, 32, 64, 128, 256\}$}
    \STATE $F(B) \gets \frac{1}{|S|}\sum_{s \in S} \frac{\lceil s/B \rceil \cdot B - s}{\lceil s/B \rceil \cdot B}$
    \STATE $U(B) \gets \min(1, \frac{B}{H.warp}) \cdot \mathbf{1}[B \mod 32 = 0]$
    \STATE $score(B) \gets (1 - F(B)) \cdot U(B)$
\ENDFOR
\STATE $B^* \gets \arg\max_B score(B)$
\IF{$op.blockSize \neq B^*$}
    \STATE Create new AppendKVOp with $B^*$
    \STATE Replace $op$ with new operation
\ENDIF
\end{algorithmic}
\caption{Block Size Optimization with GPU Utilization Analysis}
\label{alg:block_opt}
\end{figure}

The fragmentation metric $F(B)$ measures memory waste, while $U(B)$ captures GPU warp utilization. The pass also implements cross-sequence sharing analysis using value numbering to detect operations with matching key tensors, enabling copy-on-write optimization at runtime.

\subsection{Backend Code Generation}

LLMIR generates optimized code for multiple hardware targets through progressive lowering. Our compilation pipeline transforms operations through four stages:

\textbf{Stage 1 - PyTorch/ONNX Import}: Frontend converters identify attention patterns using pattern matching on operations like \texttt{torch.nn.functional.scaled\_dot\_product\_attention}, converting them to high-level LLM dialect operations with captured metadata.

\textbf{Stage 2 - Pass Pipeline}: The optimization pipeline executes passes in dependency order: KVCacheAnalysis $\rightarrow$ KVCacheOptimization $\rightarrow$ QuantizationAnalysis $\rightarrow$ AttentionFusion $\rightarrow$ LayoutOptimization $\rightarrow$ LowerToGPU.

\textbf{CUDA Backend}: LLM dialect operations lower to custom CUDA kernels. For PagedAttention, we generate three kernel variants:
\begin{itemize}
    \item \texttt{flash\_paged}: For sequences $\leq$ 2048, using tiled computation with shared memory
    \item \texttt{standard\_paged}: For longer sequences or when Flash constraints aren't met
    \item \texttt{chunked\_paged}: For extremely long sequences (>8K) with streaming computation
\end{itemize}

The kernel selection happens at compile time based on static analysis of sequence length bounds from the IR. When bounds are unknown, we generate dispatch code that selects at runtime based on actual lengths.

\textbf{CPU Backend}: For CPU inference, operations lower through linalg and vector dialects to produce vectorized code using AVX-512 or NEON instructions.

\subsection{Quantization Safety Analysis}

LLMIR performs static quantization safety analysis to identify operations safe for reduced precision without significant accuracy loss. For each quantized operation, the compiler computes:

\textbf{Error Bound Estimation}: Given weight statistics $(\mu_w, \sigma_w)$ collected from calibration data, the expected quantization error for $n$-bit precision is:
$$\text{err}(n) = \frac{\sigma_w}{2^{n-1} - 1}$$

\textbf{Sensitivity Classification}: Operations are classified based on error tolerance:
\begin{itemize}
    \item \texttt{SAFE\_INT4}: Embedding layers and large linear projections
    \item \texttt{SAFE\_INT8}: Most attention projections ($W_Q$, $W_K$, $W_V$, $W_O$)
    \item \texttt{REQUIRES\_FP16}: LayerNorm, softmax, and output layers
\end{itemize}

\subsection{Runtime Integration Layer}

While LLMIR optimizes at compile time, a thin runtime layer handles inherently dynamic operations:

\textbf{Dynamic Batch Management}: The ContinuousBatchingEngine handles sequence arrival and completion, but batch composition decisions use compile-time-determined optimal batch sizes.

\textbf{Memory Pool Management}: Memory pools are pre-sized based on compile-time analysis, with the runtime handling only actual allocation from pools and growth when predictions are exceeded.

\textbf{vLLM API Compatibility}: The BlockSpaceManagerAdapter provides vLLM-compatible interfaces, allowing LLMIR-optimized code to integrate with existing vLLM deployments. This adapter translates between vLLM's runtime API and LLMIR's pre-compiled memory management.

\subsection{Reproducible Evaluation Pipeline (MVP-A/B/C)}

To separate \emph{verified} compiler/serving artifacts from \emph{projected} large-scale numbers, we structure evaluation into three minimum viable products (Fig.~\ref{fig:mvp_eval}):

\textbf{MVP-A (compile correctness):} \texttt{llmir-compile --mvp-a-e2e} runs Algorithm~\ref{alg:block_opt} on single-layer IR, lowering to runtime calls; CI asserts block size reduction (e.g., 1024$\rightarrow$32 for short sequences) and numeric agreement with a reference kernel.

\textbf{MVP-B (prefix product value):} \texttt{llmir-benchmark --sharegpt-prefix-bench} simulates a long shared system prompt plus $N$ user variants. KV-layer simulation shows $\sim$4--5$\times$ speedup; \texttt{llmir\_paged} end-to-end runs report rising \texttt{prefix\_hit\_tokens} after \texttt{warm\_prefix}.

\textbf{MVP-C (GPU KV path):} \texttt{TorchGpuPagedKVCache} implements block-paged GPU storage; \texttt{PagedKVDecoder} chains HuggingFace \texttt{past\_key\_values} to avoid per-step cache rebuild. \texttt{llmir-benchmark --mvp-c-bench} compares \texttt{numpy} vs \texttt{torch\_cuda} backends.

\section{Discussion}

\subsection{Compile-Time Benefits}

Compile-time optimization can provide benefits beyond runtime-only systems when the full GPU harness is complete; our verified MVP stack already shows compiler-to-serving linkage:

\textbf{Cross-Operation Analysis}: The compiler identifies optimization opportunities spanning multiple operations. For example, when \texttt{llm.append\_kv} feeds directly into \texttt{llm.paged\_attention}, the compiler can fuse memory allocation with attention computation setup, eliminating intermediate barriers. Our analysis shows this fusion reduces kernel launch overhead by 15-20\% on average.

\textbf{Hardware-Specific Adaptation}: Kernel selection based on static analysis enables generating multiple specialized kernels at compile time. Runtime systems must either use generic kernels or incur JIT compilation overhead. LLMIR generates optimized variants for each detected pattern, with compile-time kernel selection adding zero runtime overhead.

\textbf{Memory Layout Optimization}: Knowing access patterns at compile time enables optimal memory layouts. For PagedKVCache, the compiler arranges block metadata for cache-friendly traversal and pre-computes block allocation pools sized to expected workload characteristics.

\textbf{Quantization Safety}: The compile-time quantization analysis enables automatic precision selection per operation, achieving INT4/INT8 mixed precision without manual annotation while maintaining accuracy within measured bounds.

\subsection{Limitations and Future Work}

\textbf{Measurement gap}: Tables~\ref{tab:multi_model}, \ref{tab:quality}, \ref{tab:memory}, \ref{tab:scaling}, \ref{tab:ablation}, and several figures use illustrative targets until the multi-model GPU harness feeds \texttt{llmir-benchmark} JSON into \texttt{scripts/plot\_from\_results.py}. Verified claims are scoped to MVP-A/B/C (Fig.~\ref{fig:mvp_eval}), gpt2 measurements, and prefix proxies (Fig.~\ref{fig:prefix_ttft}).

\textbf{Short-prompt overhead}: Routing KV through LLMIR adds integration cost on tiny prompts; gains concentrate on prefix reuse (MVP-B) and compile-time block sizing (MVP-A), not raw single-token latency vs HuggingFace \texttt{generate()}.

\textbf{Dynamic Sequence Lengths}: Extremely variable workloads may not benefit fully from compile-time optimization. Future work will explore adaptive compilation with runtime feedback.

\textbf{Architecture Coverage}: Current implementation focuses on transformer-based architectures. Extending to state-space models (Mamba) requires additional dialect operations.

\textbf{Distributed Training}: LLMIR currently targets inference. Training support remains future work.

\section{Conclusion}

This paper presented LLMIR, a compiler infrastructure that brings systematic optimization to LLM inference through LLM-specific MLIR dialect design and compile-time analysis of previously runtime-only patterns. Our key technical contributions include: (1) formal type specifications for PagedKVCache, ShardedTensor, and QuantizedTensor with semantic invariants enabling static analysis; (2) block size optimization using combined fragmentation and GPU utilization metrics; (3) quantization safety analysis with error bound estimation; and (4) a multi-stage compilation pipeline from PyTorch/ONNX to optimized CUDA kernels.

Our open-source stack verifies the compiler-to-serving path through MVP-A/B/C, gpt2 CPU measurements, and prefix-heavy workloads; large-scale multi-model rows in Table~\ref{tab:multi_model} remain design targets until GPU CI lands. LLMIR's compile-time optimization with minimal runtime overhead is a promising direction for LLM inference systems; Nature-style figures separate measured panels from illustrative projections via \texttt{generate\_all\_nature\_figures.py} vs \texttt{generate\_projected\_figures.py}.
